# 集成方案

## 1. 集成策略概述

### 1.1 集成原则
- **渐进式集成**: 完成一个模块集成一个模块
- **自动化测试**: 每个模块必须有单元测试和集成测试
- **接口契约**: 严格遵循 API 规范，确保前后端无缝对接
- **版本管理**: 使用 Git 分支策略，主分支保持稳定

### 1.2 集成流程
```
模块开发完成
    ↓
代码审查 (Code Review)
    ↓
单元测试通过
    ↓
合并到开发分支 (develop)
    ↓
集成测试
    ↓
合并到主分支 (main)
    ↓
发布
```

---

## 2. 分支管理策略

### 2.1 Git Flow 工作流

```
main (主分支，始终稳定)
  ↑
  └── develop (开发分支，集成测试)
        ↑
        ├── feature/task-1.1-auth (功能分支)
        ├── feature/task-1.2-user (功能分支)
        ├── feature/task-1.3-role (功能分支)
        └── ...
```

### 2.2 分支命名规范
- **主分支**: `main` - 生产环境代码
- **开发分支**: `develop` - 集成测试分支
- **功能分支**: `feature/{task-id}-{description}` - 如 `feature/task-1.1-auth`
- **Bug修复分支**: `fix/{bug-id}-{description}`
- **发布分支**: `release/v{version}`

### 2.3 合并规则
1. 功能分支 → develop 分支：需要 PR (Pull Request) + Code Review
2. develop 分支 → main 分支：需要集成测试通过 + 发布审批
3. 禁止直接提交到 main 分支

## 4. 集成测试规范

### 4.1 测试金字塔
```
        /\
       /E2E\        端到端测试 (少量)
      /------\
     /集成测试\     集成测试 (适中)
    /----------\
   /  单元测试  \    单元测试 (大量)
  /--------------\
```

### 4.2 单元测试
**测试范围**:
- Service 层业务逻辑
- 工具函数
- 数据验证

**后端测试框架**: pytest
**前端测试框架**: Vitest

**代码覆盖率要求**: ≥ 70%

### 4.3 集成测试
**测试范围**:
- API 接口端到端测试
- 数据库操作测试
- 前后端对接测试

**测试工具**:
- 后端: pytest + httpx
- 前端: Playwright / Cypress

### 4.4 测试环境
```
开发环境 (dev):
- 数据库: PostgreSQL (本地 Docker)
- Redis: 本地 Docker
- 前端: 本地开发服务器

测试环境 (test):
- 数据库: PostgreSQL (独立测试实例)
- Redis: 独立测试实例
- 数据: 测试数据，每次测试前重置

预发布环境 (staging):
- 与生产环境配置一致
- 使用生产数据快照
- 最终验收测试
```

## 5. 接口契约管理

### 5.1 API 文档
使用 Swagger/OpenAPI 自动生成 API 文档：
- 访问地址: `http://localhost:8000/docs`
- 所有接口必须有详细的注释
- 包含请求/响应示例

### 5.2 接口版本管理
- API 路径包含版本号: `/api/v1/`
- 不兼容变更需升级版本号: `/api/v2/`
- 旧版本保留至少 3 个月

### 5.3 Mock 数据
开发期间使用 Mock 数据解耦前后端：
- 后端: 提供 Mock API 或使用 Swagger UI 测试
- 前端: 使用 Mock Service Worker (MSW) 或本地 Mock 数据

## 6. 数据迁移策略

### 6.1 Alembic 迁移流程
```
1. 开发人员修改 models/ 下的模型文件
2. 生成迁移脚本: alembic revision --autogenerate -m "description"
3. 检查并修改迁移脚本
4. 执行迁移: alembic upgrade head
5. 提交迁移脚本到 Git
```

### 6.2 迁移脚本规范
- 迁移脚本必须包含 downgrade 逻辑
- 生产环境迁移前必须在测试环境验证
- 大表修改需考虑数据备份和回滚方案

## 7. 部署集成

### 7.1 Docker Compose 部署
```yaml
# 开发环境
docker-compose -f docker-compose.dev.yml up -d

# 测试环境
docker-compose -f docker-compose.test.yml up -d

# 生产环境
docker-compose -f docker-compose.prod.yml up -d
```

### 7.2 持续集成 (CI)
使用 GitHub Actions / GitLab CI 自动化：
- 代码提交自动运行单元测试
- PR 自动运行集成测试
- 合并到 main 自动构建 Docker 镜像
- 自动部署到测试环境

### 7.3 配置管理
不同环境使用不同的配置文件：
- `.env.development` - 开发环境
- `.env.test` - 测试环境
- `.env.production` - 生产环境
- 敏感信息使用环境变量，不提交到 Git

## 8. 问题排查与回滚

### 8.1 问题排查清单
- [ ] 检查日志输出
- [ ] 检查数据库连接
- [ ] 检查 Redis 连接
- [ ] 检查 API 接口响应
- [ ] 检查前端网络请求
- [ ] 复现问题步骤
- [ ] 确认是代码问题还是环境问题

### 8.2 回滚策略
**代码回滚**:
```bash
# 回滚到上一个稳定版本
git checkout main
git reset --hard <last-stable-commit>
git push --force origin main
```

**数据库回滚**:
```bash
# 回滚数据库迁移
alembic downgrade -1
```

**Docker 回滚**:
```bash
# 使用上一个版本的镜像
docker-compose down
docker-compose pull
docker-compose up -d
```
